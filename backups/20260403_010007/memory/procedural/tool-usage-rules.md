# 文件编辑工具使用铁律 - 【绝对禁止】

## 🚫 核心原则：`edit` 工具默认禁用

**除非用户明确说"用edit工具"或"edit这个文件"，否则`edit`视为不存在！**

---

## 📋 工具选择决策树

### 场景1：结构化配置文件（JSON/YAML/Shell等）
**例如：** `jobs.json`, `.env`, `.sh` 脚本

**操作：** `write`（全量覆盖）
- ✅ 正确：读取原文件 → 修改内容 → `write`整个新内容
- ❌ 禁止：`edit`（容易匹配失败）
- ❌ 禁止：`append`（JSON格式会坏）

**原因：** 配置文件是结构化的，整体重写最安全

---

### 场景2：文档类文件（Markdown/文本/Feishu文档）
**例如：** `MEMORY.md`, `2026-03-10.md`, 飞书文档

**操作：** `append` 或 `insert`（追加/插入）
- ✅ 正确：在末尾`append`新内容
- ✅ 正确：用`insert`在特定位置添加
- ❌ 禁止：`write`（会覆盖原文）
- ❌ 禁止：`edit`（容易匹配失败）

**原因：** 文档内容要保留历史，追加模式最安全

---

### 场景3：用户明确指令
**例如：** "把第3行的'ABC'改成'XYZ'"

**操作：** `edit`（仅限用户明确要求时）
- ✅ 正确：用户明确说"edit"或"修改这行"
- ⚠️ 注意：必须100%匹配原文，包括空格

---

## 🔒 自我约束机制

**操作前强制检查：**
```
1. 这是什么类型的文件？
   ├── 配置文件(JSON/YAML等) → 用 write
   ├── 文档(Markdown等) → 用 append/insert
   └── 不知道 → 问用户

2. 用户是否明确说"edit"？
   ├── 是 → 可以用 edit（但要小心）
   └── 否 → 绝对不用 edit

3. 用 write 前检查：
   ├── 是否确认覆盖？（配置文件中yes）
   └── 是否备份了重要内容？
```

## ✅ 正确示例

### 示例1：修改cron配置
```python
# 1. 读取原文件
read jobs.json

# 2. 在内存中修改数据
new_jobs = old_jobs + [new_task]

# 3. write整个新内容（不是edit！）
write jobs.json (完整新JSON)
```

### 示例2：添加记忆记录
```python
# 直接append到末尾
append 2026-03-10.md "\n## 新内容..."
```

### 示例3：飞书文档更新
```python

feishu_doc append doc_token content
```
