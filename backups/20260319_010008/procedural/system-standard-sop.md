## 第一章：文件操作规范

### 1.1 创建文件前检查清单

**Step 1: 确定文件类型和目录**
```
日记/日志 → memory/
流程规范 → procedural/
知识记录 → semantic/
调研资料 → research/
临时文件 → /tmp/
飞书文档 → 直接用 feishu_doc（禁止本地先写！）
```

**Step 2: 明确说出完整路径**
创建前必须向用户宣告：
> "我将创建文件：`/root/.openclaw/workspace/{目录}/文件名.md`"

**Step 3: 确认工具选择**
- 本地文件 → `write` 工具（整文件重写）
- 飞书文档 → `feishu_doc` 工具（指定 folder_token）
- ❌ **禁用 `edit` 工具**（失败率高，用户体验差）

### 1.2 禁止清单

| 禁止行为 | 正确做法 |
|---------|---------|
| `write file_path="文件名.md"` | `write file_path="/root/.openclaw/workspace/memory/文件名.md"` |
| 在 workspace 根目录创建 .md 文件 | 必须放入 memory/procedural/semantic/research/ |
| 飞书文档先在本地写 | 直接用 `feishu_doc` 在「虾米🦞」文件夹创建 |
| 使用 `edit` 工具修改文件 | 使用 `write` 工具重写整个文件 |
| 默默执行后报错 | 执行前宣告路径和预期结果 |

### 1.3 Edit vs Write 决策树

| 工具 | 成功率 | 适用场景 |
|------|--------|----------|
| edit | 60% | 简单替换，格式完美匹配 |
| write | 99% | 所有场景，特别是复杂文件 |

**决策规则：**
- 文件 < 50行 → 可以用edit，但write更保险
- 文件 >= 50行 → 必须用write
- 有格式要求（markdown表格等） → 必须用write
- 不确定 → 直接用write

**结论：默认用write，不用edit**

## 第二章：定时任务规范

### 2.1 触发条件识别

当吕总发布以下类型的任务时：
- "每天X点..."
- "每周X..."
- "定时..."
- "提醒..."
- "以后每天/每周..."
- 任何包含明确时间点的重复性任务

### 2.2 处理流程

**步骤1：识别任务类型**
- **一次性提醒**：使用 `cron add --at`
- **周期性任务**：使用 OpenClaw Gateway cron 系统

**步骤2：创建任务**
```json
{
  "name": "任务名称",
  "schedule": {"kind": "cron", "expr": "0 8 * * *", "tz": "Asia/Shanghai"},
  "sessionTarget": "isolated",
  "payload": {"kind": "agentTurn", "message": "执行内容"},
  "delivery": {"mode": "announce", "channel": "feishu", "to": "user:ou_xxx"}
}
```

**步骤3：验证**
```bash
openclaw gateway status
```

### 2.3 投递目标格式铁律

| ❌ 错误 | ✅ 正确 |
|--------|--------|
| `"to": "ou_xxx"` | `"to": "user:ou_xxx"` |
| `"to": "oc_xxx"` | `"to": "chat:oc_xxx"` |

## 第三章：工作原则

### 3.1 核心原则：自主解决问题

**遇到问题时的处理流程：**
1. **先尝试解决** - 查文档、查历史记录、试方案
2. **记录问题和尝试** - 把尝试过程和结果记下来
3. **实在解决不了再汇报** - 汇报时带上已尝试的方案和结果

**禁止行为：**
- ❌ 一遇到问题就问吕总
- ❌ 把决策责任推给吕总
- ❌ 只提问题不给方案

**应该做的：**
- ✅ 主动查找历史记录和知识库
- ✅ 给出2-3个可行方案，说明优缺点
- ✅ 小问题直接决定，大问题给出建议让吕总拍板

### 3.2 具体场景

| 场景 | 自主处理 |
|------|---------|
| 早报生成 | 自己读取历史记录、提取待办事项 |
| 定时任务 | 自己配置cron、自己测试验证 |
| 技能安装 | 自己搜索skill、阅读SKILL.md |
| 文档操作 | 自己确认格式和内容 |

### 3.3 汇报原则

**只汇报：**
- 最终完成结果
- 严重错误和事故
- 需要吕总决策的选项（带方案）

**不汇报：**
- 中间过程细节
- 已解决的小问题
- 可以自行决定的事项

## 第五章：早报生成规范

### 5.1 执行流程

1. **读取历史记录** - memory/YYYY-MM-DD.md
2. **提取待办** - 从飞书待办表读取真实数据
3. **生成内容** - 按格式生成
4. **发送** - 7:50准时发送

### 5.2 数据来源（优先级）

1. **首要来源：** 飞书待办追踪表
2. **次要来源：** memory/quitting-smoking.md（戒烟天数）
3. **第三来源：** cron jobs配置（未来48小时提醒）

### 5.3 相对日期解析规则

**使用截止日期字段判断时间，不依赖事项名称中的相对日期：**

| 截止日期 | 显示格式 |
|---------|---------|
| = 今天 | "今天 HH:MM" |
| = 明天 | "明天 HH:MM" |
| = 后天 | "后天 HH:MM" |
| 其他 | "M月D日 HH:MM" |

### 5.4 铁律

- ✅ 生成早报前**必须**读取飞书待办表
- ✅ 戒烟天数必须准确
- ✅ 时间判断必须用截止日期字段
- ❌ **禁止**虚构待办事项
- ❌ **禁止**询问吕总已经记录在表中的信息

## 第六章：错误诊断命令

### 6.1 文件编辑问题

```bash
# 查看隐藏字符
cat -A <文件>

# 转换换行符（Windows -> Unix）
sed -i 's/\r$//' <文件>

# 删除行尾空格
sed -i 's/[[:space:]]*$//' <文件>
```

### 6.2 系统状态检查

```bash
# 检查 OpenClaw 状态
openclaw gateway status

# 查看定时任务
openclaw cron list

# 检查端口
ss -tlnp | grep -E "18789|8080"
```
